Repository strategy

세 저장소와 호환 버전을 명시적으로 묶습니다

기존의 단일 저장소 main/mvp/feature 설명은 실제 구조와 다릅니다. AI, Web, Blueprint는 독립 기준선을 가집니다.

  • schema/API 변경은 두 저장소 contract test와 migration을 동반
  • 통합 전 feature 브랜치의 dirty 변경을 보존하고 의도적으로 정리
  • 테스트 통과 전 main을 “발표 가능” 또는 release-ready로 선언하지 않음
  • 기존 ver.0, ver.1 tag는 이동·덮어쓰기·삭제하지 않음
  • 새 release/tag와 push는 별도 승인 후 수행
