발표와 포트폴리오가 같은 사실을 다른 관점으로 사용하게 합니다

팀 성과, 개인 기여, 현재 구현, 후속 계획을 분리해 근거보다 큰 주장을 만들지 않습니다.

증거 우선 규칙

주장필요 근거
구현 완료기준선 소스와 테스트 또는 동적 실행 결과
운영 동작환경·시각·모드·최종 상태가 포함된 실행 로그
DB 일치운영 DB와 버전 관리된 DDL·스냅샷·ERD 대조
모델 성능데이터 분할, 기준선, 평가 지표, 재현 가능한 결과
개인 기여본인 확인, 담당 산출물, 커밋·문서·발표 등 추적 근거

팀 발표와 개인 포트폴리오 분리

팀 발표

“우리 팀은”을 주어로 문제, 구조, 협업 결과, 전체 검증과 한계를 설명합니다. 개인별 공로를 임의 배분하지 않습니다.

개인 포트폴리오

팀 프로젝트 사실을 공통 배경으로 인용하되, 본인이 수행한 문제·행동·산출물·결과만 별도 부록에서 설명합니다.

이번 팀 기반 페이지에는 개인 역할과 기여를 넣지 않습니다.

표현 규칙

  • “파이프라인 성공”, “레코드 처리 성공”, “예측 성능”을 서로 바꾸어 쓰지 않습니다.
  • 구현·검증, 운영 로그 확인, 구현·검증 제한, 성능 미검증, MVP 제외 상태를 사용합니다.
  • 계획과 후보는 현재 기능처럼 서술하지 않습니다.
  • 분석 결과를 투자 추천·수익 보장으로 표현하지 않습니다.
  • 비밀번호, API key, DB 접속정보, 이메일 인증정보, 실제 사용자 데이터는 문서에 넣지 않습니다.

갱신 규칙

  1. AI·Web 기준 커밋과 스냅샷 날짜 확인
  2. 구현·스키마·API·검증 근거의 변경점 대조
  3. 영향받는 페이지와 README를 함께 갱신
  4. 내부 링크·HTML 구조·변경 파일 범위를 검사
  5. 검토와 승인 후에만 커밋·배포 상태를 변경

KDT 1차 팀 프로젝트의 교육용 최종 MVP 기록입니다. 투자 판단이나 상용 운영 준비 완료를 의미하지 않습니다.
